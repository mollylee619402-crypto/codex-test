# codex-test

一个用于练习协作开发与工程化流程的示例仓库。

## 1. 项目简介

本项目当前处于初始化阶段，目标是为新同学提供一个“从 0 到 1”熟悉代码库的实践环境。你可以在这里学习：

- 如何阅读与维护项目文档。
- 如何搭建基础目录结构与开发流程。
- 如何通过小步提交推进功能开发。

> 当前仓库仍是最小版本，后续会逐步补充代码、测试和自动化配置。

---

## 2. 目录结构

当前目录较精简，建议按如下结构逐步演进：

```text
codex-test/
├── README.md                # 项目说明（你正在阅读）
├── src/                     # 源代码目录（后续新增）
├── tests/                   # 测试目录（后续新增）
├── docs/                    # 设计文档与规范（后续新增）
└── scripts/                 # 开发/发布脚本（后续新增）
```

目录说明建议：

- `src/`：放核心业务代码。
- `tests/`：放单元测试、集成测试。
- `docs/`：放架构设计、开发规范、FAQ。
- `scripts/`：放一键运行、检查、发布相关脚本。

## 3. 如何开始

### 3.1 克隆仓库

```bash
git clone <your-repo-url>
cd codex-test
```

### 3.2 创建开发分支

```bash
git checkout -b feat/your-topic
```

### 3.3 开发建议流程

1. 先更新/补充文档（README、docs）。
2. 再新增最小可运行代码（MVP）。
3. 补充测试与检查脚本。
4. 提交变更并发起 PR。

### 3.4 提交示例

```bash
git add .
git commit -m "docs: initialize beginner-friendly README"
```

## 5. 给 Codex 使用的注意事项

当你使用 Codex 协助本仓库开发时，建议遵循：

1. **先读文档再改代码**
   - 优先让 Codex 阅读 `README.md` 和 `docs/`，确认项目约束后再修改。

2. **一次只做一类改动**
   - 一个任务聚焦一件事（例如“仅补 README”或“仅新增测试”），便于审查与回滚。

3. **要求输出变更摘要**
   - 让 Codex 在结果中写清：改了哪些文件、为什么改、如何验证。

4. **要求执行最小验证命令**
   - 例如：格式检查、测试命令、构建命令。

5. **避免高风险批量重构**
   - 在项目早期，优先小步快跑；大改动应先提交设计说明。

6. **提交前人工复核**
   - 尤其关注：删除文件、依赖升级、脚本改动、配置文件变更。
